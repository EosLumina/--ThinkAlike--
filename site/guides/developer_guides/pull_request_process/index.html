Pull Request Process Guide

1. Introduction

This document outlines the standard process for submitting Pull Requests (PRs) to the ThinkAlike project. Following these guidelines ensures code quality, consistency, and clear communication between contributors and maintainers. Whether you're a core team member or an external contributor, this process applies to all code changes.

2. Before Creating a Pull Request

2.1 Preparation

  1. Create an Issue First (if one doesn't exist):

  2. All code changes should be tied to an issue in the GitHub tracker

  3. Follow the Issue Labels Guide for proper categorization

  4. Get feedback on your proposed approach before writing code

  5. Fork and Branch:

  6. Fork the repository if you're an external contributor

  7. Create a feature branch from main with a descriptive name:

    git checkout -b feature/add-user-preferences
# or

git checkout -b fix/login-validation-error

  8. Use prefixes like feature/, fix/, docs/, refactor/, etc.

  9. Keep Changes Focused:

  10. Each PR should address a single issue or feature

  11. Avoid combining unrelated changes in a single PR

  12. For large features, consider breaking them into smaller PRs

2.2 Development Standards

  1. Follow Code Style:

  2. Adhere to the project's coding standards for:

    • Python (backend): Follow PEP 8

    • TypeScript/JavaScript (frontend): Follow ESLint/Prettier config

  3. Run linters before committing:

    # Backend

flake8 app/
black app/

# Frontend

npm run lint

  4. Write Tests:

  5. Include appropriate tests for your changes:

    • Unit tests for core logic

    • Integration tests for API endpoints

    • E2E tests for critical user flows

  6. Aim for at least 80% test coverage for new code

  7. Ensure all existing tests pass

  8. Document Your Changes:

  9. Update relevant documentation

  10. Add comments for complex logic

  11. Include docstrings for new functions/methods

3. Creating the Pull Request

3.1 Commit Guidelines

  1. Write Meaningful Commit Messages:

  2. Use the imperative mood: "Add feature" not "Added feature"

  3. Format: [TYPE]: Short summary (50 chars or less)

  4. Examples:

    feat: add user preference settings
fix: resolve authentication token expiration bug
docs: update API documentation
refactor: simplify matching algorithm

  5. For larger changes, include a body that explains what and why (not how)

  6. Organize Commits:

  7. Make logical, atomic commits

  8. Consider squashing fixup commits before creating PR

  9. Rebase to keep a clean history:

    git fetch origin
git rebase origin/main

3.2 PR Submission

  1. Create the PR on GitHub:

  2. Go to the repository on GitHub

  3. Click "New pull request"

  4. Select your branch

  5. Fill in the PR template completely

  6. PR Template Contents:

  7. Link to the related issue(s)

  8. Clear description of changes

  9. Screenshots/videos for UI changes

  10. Checklist of completed items

  11. Notes on testing methodology

  12. Any deployment considerations

  13. Mark as Draft if needed:

  14. Use GitHub's "Draft PR" feature if work is still in progress

  15. Convert to ready when you want review

4. PR Review Process

4.1 Request Reviews

  1. Assign Reviewers:

  2. Request reviews from appropriate team members

  3. At least one core maintainer should review each PR

  4. Consider requesting specialist reviews for complex areas

  5. Respond to Feedback:

  6. Address all comments and suggestions

  7. Explain your reasoning if you disagree with feedback

  8. Make requested changes promptly

  9. Mark conversations as resolved after addressing them

4.2 CI/CD Checks

  1. Verify Automated Checks:

  2. Ensure all CI pipelines pass:

    • Tests (unit, integration, E2E)

    • Linting

    • Type checking

    • Security scans

  3. Fix any failing checks before requesting re-review

  4. Update Based on Feedback:

  5. Push new commits to address review comments

  6. Consider using git commit --fixup for small changes

  7. Rebase and squash fixups when ready for final review:

    git rebase -i --autosquash origin/main

5. Merging

  1. Merge Requirements:

  2. At least one approval from a core maintainer

  3. All CI checks pass

  4. All discussions resolved

  5. PR is up-to-date with the main branch

  6. Merge Responsibility:

  7. Core team members typically handle the actual merge

  8. Use the appropriate merge strategy:

    • Squash and merge: For most feature PRs (creates a single commit)

    • Rebase and merge: For PRs with well-structured commits worth preserving

    • Merge commit: Rarely used, only for major features with extensive history

  9. After Merging:

  10. Delete the feature branch

  11. Close the associated issue or update its status

  12. Deploy if required (following the deployment process)

6. Special Cases

6.1 Hotfixes

For urgent production issues:

  1. Create branch directly from the production tag: hotfix/critical-auth-issue
  2. Follow an expedited review process
  3. Merge to both main and the appropriate release branch
  4. Deploy as soon as possible following approval

6.2 Long-Running Feature Branches

For major features that take weeks to develop:

  1. Rebase regularly against main to prevent major conflicts
  2. Consider creating intermediate PRs for reviewable chunks
  3. Use feature flags to merge code that isn't ready for activation

7. Tips for Successful PRs

  • Keep PRs Small: Aim for <500 lines changed when possible

  • Communicate: Use PR comments to explain decisions and ask questions

  • Be Patient: Understand that review takes time, especially for complex changes

  • Be Responsive: Address feedback promptly to keep the PR moving

  • Learn from Feedback: Use review comments as learning opportunities

By following this pull request process, we maintain high code quality while allowing for efficient collaboration among all ThinkAlike contributors.
